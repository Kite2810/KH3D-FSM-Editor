Kingdom Hearts 3D: Dream Drop Distance — file formats FSM (Finite State Machine)

.fsm files describe the AI brain of an entity in Kingdom Hearts 3D: Dream Drop Distance. Each file is a finite state machine: a set of states (Idle, Pursuit, Attack1…), the transitions between them, and the conditions that gate each transition. The engine sits in one state, evaluates that state's outgoing transitions every tick, and moves to the first state whose conditions are all satisfied.

They are found alongside the other per-object files (e.g. m_di020.fsm for a companion, m_de120.fsm / b_de020.fsm for enemies and bosses). The format is a flat binary identified by the ASCII magic @FSM; everything is little-endian.

This document describes version 3 (the value at offset 0x04), the only version observed so far. It was reverse-engineered from companion, enemy and boss files and validated against m_di020 · 4,896 B m_de120 · 17.5 KB m_de490 · 18 KB b_de020 · 26.9 KB.

File layout

The file is four regions back-to-back. Cross-references between regions are stored as raw file offsets (not indices), so nothing can be moved without re-pointing everything that referenced it.

RegionLocationDescription
Header0x00 – 0x0CMagic, version, and the offset where the symbol table ends.
Symbol table0x0C – dataOffsetNamed strings — every state and condition variable. Referenced by byte offset.
Record tablescannedOne 80-byte record per outgoing transition, grouped by owning state.
Entry / detail datapointed-toThe transition entries and their condition lists, reached through pointers.

Header

The first 12 bytes.

OffsetTypeDescription
0x00char[4]Magic — always @FSM.
0x04u16Version. Observed value: 3.
0x06u16Unknown (flags?).
0x08u32Data offset — the byte where the symbol table ends. The symbol table occupies 0x0C up to this value, so symbolCount = (dataOffset − 0x0C) / 36.

Symbol table

An array of fixed 36-byte entries running from 0x0C to the data offset. Every string the file needs lives here exactly once — both state names and condition variable names. Everywhere else in the file, a symbol is referenced by the byte offset of its name field.

OffsetTypeDescription
0x00char[32]Symbol name, ASCII, null-padded.
0x20u32Symbol id (a hash/index; not needed to resolve references).

The format does not tag whether a symbol is a state or a condition. That is derived structurally — see Evaluation & editing.

Record table

An array of 80-byte records. Each record is one outgoing transition; a state that has several transitions owns several consecutive records. Only three fields are needed to walk the machine — the rest of each record is currently unknown / padding.

OffsetTypeDescription
0x04u16Record index.
0x08u32Entry pointer — file offset of this transition's entry block.
0x0Cu32Owner name offset — the symbol offset of the state that owns this transition.
…—Remaining bytes: unknown / padding.

Locating the table. The record table is not at a fixed offset. Locate it by scanning forward from the end of the symbol table for the longest run of consecutive 80-byte records whose owner field (+0x0C) is a valid symbol offset.

Transition entry

Pointed to by a record's entry pointer (record + 0x08). It names the destination state and points at the condition list.

OffsetTypeDescription
0x00u32Target pointer — a file offset pointing at the owner-name field (+0x0C) of the destination state's record. Dereference it to get that state's symbol offset, hence its name.
0x04u16Tag / flags (unknown).
0x06u16Condition count.
0x08u32Detail pointer — file offset of the condition detail array.

Because the target is reached by pointing at a state's record, re-targeting a transition is just overwriting this one u32 with the owner-field offset of a different state — no bytes move.

Condition detail

An array of conditionCount entries, 12 bytes each, at the entry's detail pointer. A transition fires only when all of its conditions hold (logical AND); a count of zero means the transition always fires.

OffsetTypeDescription
0x00u32Condition-name offset — the symbol offset of the variable being tested (e.g. Probability, TargetDistance).
0x04u32Operator code.
0x08u32Parameter value. Its interpretation depends on the operator and variable.
 
// the chain that resolves one transition
record +0x08  →  entry
                 ├─ +0x00  →  destination state record   (target)
                 ├─ +0x06  =  condition count
                 └─ +0x08  →  detail[0 .. count]
                              ├─ +0x00  →  variable name   (symbol)
                              ├─ +0x04  =  operator
                              └─ +0x08  =  value

Operator codes

The u32 at detail + 0x04. A comparison operator carries a numeric threshold; is carries a 0/1 flag. Labels are best-effort.

CodeOperatorNotes
0x21isBoolean / equality. Value is a 0/1 flag.
0x01<Less-than. Numeric threshold.
0x03<=Less-than-or-equal.
0x04<Alternate less-than seen in the wild.
0x05>Greater-than.
0x06>=Greater-than-or-equal.

Parameter kinds

The parameter is a raw 32-bit word; how to read it is inferred from the operator and the variable rather than stored explicitly.

KindWhenRead as
boolis (0x21) with a raw value of 0 or 1.Flag (false / true).

Evaluation & editing

How a state ticks

While in a state, the engine evaluates that state's outgoing transitions in order and takes the first whose conditions all pass. Idle is the universal fallback / follow state; pointing transitions at Idle effectively strips an entity's native behaviour and pins it there.

States vs. conditions

Since symbols are untyped, classification is structural: a symbol used as a record owner or as a transition target is a state; any other symbol referenced as a condition name is a condition variable. This reproduces the correct split on every file tested and generalises to names never seen before.

In-place editing

Four things a transition expresses — its target, and each condition's variable, operator and value — are all single u32 fields holding either a symbol offset or a literal. Each can be overwritten with another valid value of the same width, so no pointer ever moves and the file cannot be structurally corrupted. Adding or removing transitions/conditions, by contrast, changes counts and array sizes and would shift every following offset.

This size-preserving property is what makes safe editing practical: retarget a transition, swap a condition's variable or operator, or tweak a threshold — all as 4-byte patches in place.
